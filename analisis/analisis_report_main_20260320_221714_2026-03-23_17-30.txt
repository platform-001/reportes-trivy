En este documento, se enumeran 62 problemas de seguridad de código abierto relevantes para el paquete logback en Log4J. Se incluyen detalles sobre los problemas de seguridad, su severidad y impacto, así como las soluciones proporcionadas.

1. CVE-2023-6378 - Serialization vulnerability in logback receiver
   Severity: High
   Impact: DoS Condition
   Paquete: com.fasterxml.jackson.core:jackson-core
   Solución: Number Length Constraint Bypass in Async Parser Leads to Potential DoS Condition

En el paquete logback, se ha descubierto una vulnerabilidad de serialización en la recepción del servidor logback. Se trata de un problema de seguridad crítico que puede dar lugar a un ataque DoS (Denial of Service) si se realiza una llamada de tipo AsyncParser.

La vulnerabilidad consiste en el uso de un paquete de serialización, Jackson, que intenta hacer un constrained-length constraint en los datos asociados con la lógica del servidor logback. En consecuencia, se pueden llevar a cabo operaciones no validamente asignadas y la operación AsyncParser puede producirse sin el cumplimiento de las restricciones posibles.

Con esta vulnerabilidad, un atacante podría obtener acceso al sistema en el que está ejecutando y ejecutar código arbitrario, incluso si no se incluyó una directiva de seguridad apropiada para la ruta del servidor. El impacto potencial es extenso ya que esto puede conducir a un ataque de tipo DoS en las aplicaciones.

2. CVE-2025-55163 - SslHandler doesn't correctly validate packets which can lead to native crash when using native SSLEngine
   Severity: High
   Impact: Native Crash
   Paquete: org.springframework.boot:spring-boot
   Solución: SslHandler doesn't correctly validate packets which can lead to native crash when using native SSLEngine

En el paquete Spring Boot, se ha descubierto una vulnerabilidad de seguridad crítica que puede conducir a un ataque de tipo Native Crash si se produce un error en la validación de paquetes SSL. El código puede llegar a producir un error y producirse un comportamiento inesperado cuando se utiliza el mecanismo de SSL/TLS.

La vulnerabilidad consiste en el uso del método setTrustManager(TrustManager) del paquete Spring Security, que permite validar paquetes SSL, así como de otros protocolos, como HTTPS o HTTP. Se trata de un error crítico que puede conducir al ejecutamiento inesperado de código que no tiene el contexto adecuado para su uso.

Este ataque de tipo Native Crash puede producirse en cualquier aplicación, incluso si se utiliza un software de seguridad compatible con la validación SSL/TLS. Si se usa Spring Boot con una aplicación que admite SSL, esto podría conducir a un ataque que afecte el sistema al menos.
